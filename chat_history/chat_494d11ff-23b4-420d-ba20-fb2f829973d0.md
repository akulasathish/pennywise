### 🧠 My Understanding of your "Penny Wise" Application

Your requirement is to build a lightweight payment reconciliation bridge for your **Property Management System (PMS)** subscription model, operating under these constraints:

1. **No Payment Gateway API:** Since you do not have a merchant/current bank account to register for official APIs like Razorpay, you cannot track payments through standard webhooks.
2. **Manual UPI / Bank Credits:** Instead, users will scan your personal UPI QR code to make bank transfers (like Rs. 100). When they pay, the money instantly goes directly to your bank account, and your bank sends a notification SMS to your phone.
3. **The Messaging Problem:** Your phone receives hundreds of normal texts. You need a system that scans your incoming SMS inbox, but triggers *only* when it detects a message from a specific bank (e.g., State Bank of India) containing specific keywords (e.g., "credited" or "received") and a valid transaction amount.
4. **The Concurrency Problem (The "Penny Wise" Core):** If Person A and Person B both scan a generic QR code for **Rs. 100** at the exact same time, your bank sends two SMS alerts saying "credited Rs. 100". There is absolutely no way to know which payment belongs to which user.
5. **The "Penny Wise" Solution:**
   * When **Person 1** starts the checkout, the system assigns them exactly **Rs. 100.00**. This amount is locked for them in the database for 10 minutes.
   * When **Person 2** initiates checkout at the same time, the system detects that Rs. 100.00 is active. It assigns Person 2 exactly **Rs. 100.01**.
   * When **Person 3** initiates checkout, they get **Rs. 100.02**.
   * If Person 2 pays, the bank sends an SMS saying: *"A/c XXXXX credited with Rs. 100.01"*. 
   * The app reads this message, extracts `100.01`, finds the matching pending record in your database, and instantly unlocks the subscription for **Person 2**.
   * Meanwhile, Person 1 gets unlocked when the SMS for `100.00` arrives. 

This is a beautiful, highly effective, zero-cost reconciliation technique!

### 💻 Showing the Solution: The Core Code

#### 1. The Penny-Wise Fractional Allocation Code
This logic (written in `database.js`) checks active, non-expired checkouts and assigns the lowest available fraction (starting from `.00` up to `.99`) to prevent conflicts:

```javascript
allocatePennyWiseAmount(userId, userName, basePrice) {
  // 1. Mark expired checkouts (older than 10 mins) as 'expired' to free up fractions
  this.cleanExpiredPayments();
  
  // 2. Get all currently active (pending) checkout requests
  const activeRequests = this.data.paymentRequests.filter(req => req.status === 'pending');
  
  // 3. Create a set of exact amounts already taken
  const allocatedExactAmounts = new Set(activeRequests.map(req => Number(req.amountExact).toFixed(2)));
  
  let targetAmount = basePrice;
  let increment = 0.01;
  let found = false;
  
  // 4. Sequential search for a free fractional amount from Rs. 100.00 to 100.99
  for (let i = 0; i < 100; i++) {
    const currentCandidate = (basePrice + (i * increment));
    const candidateStr = currentCandidate.toFixed(2);
    
    // If Rs. 100.00 is taken, try 100.01, then 100.02, etc.
    if (!allocatedExactAmounts.has(candidateStr)) {
      targetAmount = parseFloat(candidateStr);
      found = true;
      break;
    }
  }
  
  // Expiry window (e.g., 10 minutes)
  const expiryMin = this.data.settings.expiryMinutes || 10;
  
  const newRequest = {
    userId,
    userName,
    amountBase: basePrice,
    amountExact: targetAmount,
    status: "pending",
    expiresAt: Date.now() + expiryMin * 60 * 1000
  };
  
  // 5. Store the unique payment request in the DB
  return this.insert('paymentRequests', newRequest);
}
```

#### 2. Filtering incoming messages and parsing the exact amount
When your phone sends the SMS text to the server, this logic (written in `server.js`) verifies the source and extracts the exact credited rupees and paise using regular expressions:

```javascript
// Helper to extract amounts from different Bank formats
function parseSmsForAmount(body) {
  if (!body) return null;
  const text = body.toLowerCase().replace(/\s+/g, ' ');

  const patterns = [
    // Matches "Rs. 100.02" or "Rs 100.02" or "Rs.100.02"
    /(?:rs\.?|inr|rupees)\s*([0-9]+(?:\.[0-9]{1,2})?)/i,
    
    // Matches "credited with/by 100.02"
    /credited\s+(?:with|by|of)?\s*(?:rs\.?|inr)?\s*([0-9]+(?:\.[0-9]{1,2})?)/i,
    
    // Matches "received 100.02"
    /received\s+(?:rs\.?|inr)?\s*([0-9]+(?:\.[0-9]{1,2})?)/i,
    
    // Matches "deposited Rs 100.02"
    /deposited\s+(?:rs\.?|inr)?\s*([0-9]+(?:\.[0-9]{1,2})?)/i,

    // Fallback search for any decimal number
    /(?:credit|deposit|receive|ref|upi).+?\b([0-9]+\.[0-9]{2})\b/i
  ];

  for (const pattern of patterns) {
    const match = text.match(pattern);
    if (match && match[1]) {
      const amount = parseFloat(match[1]);
      if (!isNaN(amount) && amount > 0) return amount;
    }
  }
  return null;
}
```

#### 3. Matching and Activating the User Account
Once an amount is successfully parsed (e.g. `100.01`), the server finds the pending request and updates your database to activate the subscription:

```javascript
app.post('/api/sms', (req, res) => {
  const { sender, body } = req.body;
  const settings = db.getSettings();
  const normalizedSender = (sender || "UNKNOWN").toUpperCase();

  // 1. Verify if sender is allowed (e.g., "SBI-Alert", "HDFCBank")
  let senderAllowed = settings.allowedSenders.some(allowed => 
    normalizedSender.includes(allowed.toUpperCase())
  );

  // 2. Verify if the SMS contains required credit keywords
  let containsKeyword = settings.keywords.some(kw => 
    body.toLowerCase().includes(kw.toLowerCase())
  );

  if (!senderAllowed || !containsKeyword) {
    db.insert('receivedSms', { sender, body, parsedAmount: null, status: "ignored" });
    return res.json({ status: "ignored" });
  }

  // 3. Parse amount from SMS
  const amount = parseSmsForAmount(body);
  if (amount === null) {
    db.insert('receivedSms', { sender, body, parsedAmount: null, status: "unmatched" });
    return res.json({ status: "unmatched", error: "Could not parse amount" });
  }

  // 4. Find the matching active payment request (e.g. amountExact === 100.01)
  db.cleanExpiredPayments();
  const pendingRequests = db.get('paymentRequests').filter(req => req.status === 'pending');
  const match = pendingRequests.find(req => Math.abs(req.amountExact - amount) < 0.005);

  if (match) {
    // 5. Success! Complete transaction and activate PMS subscription
    db.update('paymentRequests', match.id, { status: "completed" });
    db.update('users', match.userId, { status: "active", expiresAt: Date.now() + 30*24*60*60*1000 }); // 30 days
    db.insert('receivedSms', { sender, body, parsedAmount: amount, matchedPaymentId: match.id, status: "processed" });

    return res.json({ success: true, status: "processed", matchedUser: match.userName });
  } else {
    db.insert('receivedSms', { sender, body, parsedAmount: amount, status: "unmatched" });
    return res.json({ success: true, status: "unmatched", reason: "No active checkout for this exact amount" });
  }
});
```

To prevent this, the client-facing checkout portal (`checkout.html`) automatically generates a **deep-linked UPI QR Code**.

`upi://pay?pa=yourupi@bank&pn=PMS_Subscriptions&am=100.01&cu=INR`
into a QR code, if the customer scans it using **GPay, PhonePe, or Paytm**, their app automatically locks the payee to your name, and locks the payment amount to exactly **Rs. 100.01**. They cannot edit the amount, ensuring 100% accurate, human-error-free reconciliation.
